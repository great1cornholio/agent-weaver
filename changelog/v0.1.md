## 2. Zrealizowane epiki

### Epic 1–3: Core orchestrator (session, lifecycle, pipeline)

- Session manager — tworzenie/kill/listing sesji, worktree provisioning
- Lifecycle manager — state machine + polling loop (30 s) z reakcjami
- Reactions engine — `ci-failed`, `changes-requested`, `approved-and-green`, `agent-stuck`, itd.
- Config z Zod validation, per-project overrides

### Epic 4: CLI preflight & smoke

- `ao spawn <project> <issue>` — pełny flow tworzenia sesji
- Preflight checks: binary (`glab`/`gh`/`git`), auth (tokeny), repo access
- Generyczny smoke script (`ao-smoke`)

### Epic 5: VRAM Scheduler & pipeline integration (2026-02-22)

- `VramScheduler` z queue lookahead i anti-starvation
- Integracja z `SessionManager` — `vram.slot.acquired/waiting/released`
- Config: `concurrency.queueLookahead`, `maxSkipsPerTask`, `retryBackoff`
- Persistence/restore stanu schedulera

### Epic 6: Plugin discovery & developer API (2026-02-22)

- Config-driven loading: `plugins.<slot>[]` z module string lub `{ module, config }`
- Slot mismatch guard, graceful error handling, warning telemetry
- Plugin Authoring Guide (`docs/PLUGIN_AUTHORING.md`)
- Schema-level validation, typed config contract

## 5. Co działa dobrze

- **Architektura pluginów** — wzorzec `PluginModule` z `satisfies`, zero zewnętrznych zależności, `execFile` z timeoutami (zgodność z CLAUDE.md security rules)
- **Lifecycle manager platform-agnostic** — polling + state machine niezależne od SCM, pluginy GitLab podłączone bez zmian w core
- **VRAM Scheduler** — queue lookahead, anti-starvation, persistence, integracja z session manager
- **Preflight CLI** — walidacja `glab` binary + tokenów z actionable error messages
- **Config validation** — Zod schema z `.passthrough()` poprawnie przepuszcza pola typu `url` dla self-hosted GitLab
- **Plugin discovery** — config-driven loading z slot mismatch guard i graceful degradation
- **Dog-fooding** — projekt sam siebie hostuje (`ao` projekt w `agent-orchestrator.yaml`)
